## Реестр уязвимостей

### SEC-001 — HTML-инъекция через Telegram API (Stored XSS в Telegram-чате)

| Поле | Значение |
|---|---|
| **Приоритет** | **CRITICAL** |
| **Статус** | **Исправлена** (08.02.2026) |
| **Категория** | OWASP A03:2021 — Injection |
| **Файл** | `public/api/send-telegram.php` |
| **Описание** | Поле `text` из тела запроса передавалось в Telegram API с `parse_mode: HTML` без санитизации HTML-тегов. Атакующий мог внедрить произвольные HTML-теги в сообщение Telegram-чата. |
| **Вектор атаки** | `POST /api/send-telegram.php` с телом `{"text": "<a href='https://evil.com'>Нажмите для подтверждения</a>"}` |
| **Исправление** | Эндпоинт больше не принимает поле `text`. Вместо этого принимает структурированные поля (`name`, `phone`, `message`, `service`, `contactMethods`). Все пользовательские данные санитизируются через `htmlspecialchars()` перед вставкой в шаблон сообщения. HTML-сообщение для Telegram собирается исключительно на сервере. |

### SEC-002 — Отсутствие серверной валидации входных данных

| Поле | Значение |
|---|---|
| **Приоритет** | **HIGH** |
| **Статус** | **Исправлена** (08.02.2026) |
| **Категория** | OWASP A03:2021 — Injection, A04:2021 — Insecure Design |
| **Файл** | `public/api/send-telegram.php:60-90` |
| **Описание** | Эндпоинт принимал единственное поле `text` без валидации. Клиентская валидация легко обходилась через cURL/Postman. |
| **Исправление** | Сервер теперь принимает отдельные поля (`name`, `phone`, `message`, `service`, `contactMethods`) и валидирует каждое: regex `/^\+?[\d\s\-()]{7,18}$/` для телефона, `mb_strlen` для имени (макс. 200), whitelist типов контактов (`whatsapp`, `telegram`, `max`), ограничение длины сообщения (2000 символов). |

### SEC-003 — Слабый rate limiting, уязвим к обходу через IP-спуфинг

| Поле | Значение |
|---|---|
| **Приоритет** | **HIGH** |
| **Статус** | **Исправлена** (08.02.2026) |
| **Категория** | OWASP A04:2021 — Insecure Design, DoS |
| **Файл** | `public/api/send-telegram.php:139-174` |
| **Описание** | Rate limiting был основан на `$_SERVER['REMOTE_ADDR']` с интервалом 5 секунд без ограничения запросов в час (720 запросов/час = спам). |
| **Исправление** | Интервал увеличен до 30 секунд. Добавлен лимит 10 запросов/час на IP с автоочисткой устаревших записей. Остаток времени ожидания сообщается клиенту. Рекомендуется дополнительно настроить `set_real_ip_from` в Nginx и `limit_req_zone`. |

### SEC-008 — Переменные Telegram-токена с префиксом VITE_ (утечка в клиентский бандл)

| Поле | Значение |
|---|---|
| **Приоритет** | **CRITICAL** |
| **Статус** | **Исправлена** (08.02.2026) |
| **Описание** | В `deploy-node.js` токен Telegram читался из `process.env.VITE_TELEGRAM_BOT_TOKEN`. Префикс `VITE_` означает, что Vite может инжектировать эту переменную в клиентский бандл через `import.meta.env`. |
| **Категория** | OWASP A02:2021 — Cryptographic Failures (утечка секретов) |
| **Файл** | `scripts/deploy-node.js:140-142` |
| **Исправление** | Переменные переименованы: `VITE_TELEGRAM_BOT_TOKEN` → `TELEGRAM_BOT_TOKEN`, `VITE_TELEGRAM_CHAT_ID` → `TELEGRAM_CHAT_ID`. Без префикса `VITE_` переменные не попадают в клиентский бандл. Токен в клиентском коде не используется (BUG-002 исправлен ранее — создан серверный PHP-прокси). |
