#### ~~BUG-001: Файл `.env` с секретами коммитится в Git-репозиторий~~ ✅ ИСПРАВЛЕНО
- **Серьёзность:** CRITICAL
- **Компонент:** `.env`, `.gitignore`
- **Описание:** Файл `.env` содержит:
  - `VITE_TELEGRAM_BOT_TOKEN` — токен Telegram-бота
  - `VITE_TELEGRAM_CHAT_ID` — ID чата
  - `DEPLOY_HOST` — IP сервера
  - `DEPLOY_USER` — пользователь root
  - `DEPLOY_REMOTE_DIR` — путь на сервере
  
  При этом `.env` **не добавлен в `.gitignore`**. Любой, кто имеет доступ к репозиторию, получает полный доступ к Telegram-боту и знает IP-адрес сервера с root-доступом.
- **Файл:** `.gitignore` — отсутствует строка `.env`
- **Рекомендация:** 
  1. Добавить `.env` в `.gitignore`
  2. Немедленно ротировать (перевыпустить) Telegram Bot Token
  3. Удалить `.env` из истории Git (`git filter-branch` или `BFG Repo-Cleaner`)
- **Статус:** ✅ Исправлено. `.env`, `.env.local`, `.env.production`, `.env.*.local` и `**/api/.env.php` добавлены в `.gitignore`.

#### ~~BUG-002: Telegram Bot Token попадает в клиентский бандл (утечка секрета)~~ ✅ ИСПРАВЛЕНО
- **Серьёзность:** CRITICAL
- **Компонент:** `RegistrationModal.tsx`, строки 164-165
- **Описание:** Переменные `VITE_TELEGRAM_BOT_TOKEN` и `VITE_TELEGRAM_CHAT_ID` используются с префиксом `VITE_`, что означает, что Vite **встраивает их значения прямо в JavaScript-бандл** на этапе сборки. Любой пользователь сайта может открыть DevTools → Sources и увидеть токен бота. Злоумышленник может использовать этот токен для:
  - Отправки спама от имени бота
  - Чтения сообщений в чате
  - Удаления бота
- **Рекомендация:** Вынести отправку сообщений в серверный прокси (backend API / serverless function), чтобы токен никогда не попадал на клиент.
- **Статус:** ✅ Исправлено. Создан серверный PHP-прокси `public/api/send-telegram.php`. Токен хранится в `api/.env.php` только на сервере. `RegistrationModal.tsx` теперь вызывает `/api/send-telegram.php` вместо прямого обращения к Telegram API. Деплой-скрипт автоматически создаёт `.env.php` на сервере. Верификация: `grep -r "8591376782" dist/` → токен не найден в бандле.

#### BUG-004: Файл `404.html` содержит редирект на несуществующий путь `/elena-fit-magic/`
- **Серьёзность:** HIGH
- **Компонент:** `public/404.html`, строка 6
- **Описание:** Файл `404.html` содержит `<meta http-equiv="refresh" content="0; url=/elena-fit-magic/">`. Этот путь — артефакт от GitHub Pages (где base URL был `/elena-fit-magic/`). При попадании на 404 на сервере Apache пользователь будет перенаправлен на несуществующий путь, что вызовет бесконечный цикл редиректов или ещё одну 404-ошибку.
- **Рекомендация:** Изменить редирект на `/` или удалить файл `404.html`, так как `.htaccess` уже обрабатывает SPA-роутинг.
- **Статус:** ✅ Исправлено. Редирект в `public/404.html` изменён с `/elena-fit-magic/` на `/`.

#### BUG-011: Конфликт двух систем деплоя — GitHub Pages vs SFTP

- **Компонент:** `.github/workflows/deploy.yml`, `scripts/deploy-node.js`

- **Рекомендация:** Удалить `.github/workflows/deploy.yml` если деплой идёт только через SFTP, либо обновить его для деплоя на реальный сервер.
- **Статус:** ✅ Исправлено. Файл `.github/workflows/deploy.yml` удалён — деплой идёт только через SFTP.
